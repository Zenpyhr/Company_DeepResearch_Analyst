# Next Steps

This document captures the current state of the `NVDA Analyst Agent`, the design direction we want next, and a staged roadmap for making the system more dynamic without sacrificing stability.

## In Simple Words

Right now, the system works like this:

```text
User question
-> decide what kind of question it is
-> collect the right evidence
-> analyze the evidence
-> write the final grounded answer
```

More specifically:

- The `Collector Agent` gets the evidence.
- The `EDA Agent` turns that evidence into findings.
- The `Analyst Agent` writes the final answer.
- The system also saves useful output files such as CSVs, chart specs, markdown memos, and evaluation summaries. These saved files are called **artifacts**.

After the recent fixes, the system is now better at:

- keeping revenue questions mostly numeric
- keeping market questions mostly market-focused
- pulling better risk text from filings
- keeping some press-release evidence in mixed questions
- preserving more financial history for trend analysis
- ranking the final findings in a better order

## Clean Table

| Part of the system | What it does now | Future LLM upgrade |
|---|---|---|
| Question routing | Uses simple rules to decide whether the question is financial, market, qualitative, or mixed | Let an LLM decide the best path more flexibly |
| Collector Agent | Retrieves the evidence needed for the question | Let an LLM decide retrieval order and whether more evidence is needed |
| Qualitative retrieval | Uses rule-based search and scoring over SEC filings and press releases | Add richer LLM-guided retrieval or better query rewriting |
| Quantitative retrieval | Uses fixed Python + SQL helpers to get financial and market rows | Later add a controlled Text2SQL or schema-aware planning layer |
| EDA Agent | Runs stable analysis tools over the retrieved data | Let an LLM choose which analysis tools to run |
| Financial analysis | Calculates trend changes from retrieved numeric rows | Let an LLM decide which comparisons matter most |
| Market analysis | Measures movement around reporting anchors | Let an LLM choose smarter event windows and comparisons |
| Text analysis | Uses section counts, theme dictionaries, and keyword matching | Let an LLM explain what the text really means and compare sources |
| Analyst Agent | Builds the final answer from findings and sources | Let an LLM produce stronger synthesis, tradeoffs, and clearer memo-style answers |
| Retry loop | Has one simple loop if evidence is missing or weak | Let an LLM decide when to retrieve again and what is missing |

## What Is LLM-Driven Right Now

Only a small part of the system is truly LLM-driven today.

- The LLM can help rewrite the research note.
- The LLM can add a short EDA takeaway.
- The LLM can write the final answer in more natural language.

But the core system is still mostly rule-based and deterministic:

- routing is mostly rules
- retrieval is mostly fixed logic
- SQL is not generated by the LLM
- EDA tools are deterministic
- retry logic is simple

So the best description of the current system is:

**stable deterministic pipeline + light LLM help**

## What To Do Next

The best next direction is not to make everything LLM-driven at once.

Instead, we should improve the system in this order:

1. Keep improving the current output quality.
   - Make mixed answers cleaner.
   - Make risk summaries more specific.
   - Make the final answer sound more like a real analyst note.
2. Add more LLM help in places where it is safe.
   - Better routing
   - Better qualitative interpretation
   - Better final synthesis
3. Keep the low-level parts stable.
   - database access
   - retrieval helpers
   - numeric calculations
   - artifact writing
4. Add more dynamic behavior slowly.
   - smarter tool selection
   - more intentional retry logic
   - later, maybe controlled Text2SQL

## Recommended Future LLM Layers

These are the best places to add more LLM in the future:

### 1. Better routing

Right now the system uses simple rules.

Later, the LLM can decide:

- what kind of question this is
- whether it should use text, numbers, market data, or all three
- whether the question should be broken into smaller parts

### 2. Better retrieval planning

Right now the retrieval plan is mostly fixed by the question type.

Later, the LLM can decide:

- whether to get filings first or financial rows first
- whether more evidence is needed
- whether press releases should be forced into the evidence mix

### 3. Better qualitative interpretation

Right now the text analysis is still mostly rule-based.

Later, the LLM can read the retrieved text and explain:

- what the biggest risks really are
- what management is emphasizing
- how filings differ from press releases
- what overall story the text is telling

### 4. Better final synthesis

This is one of the safest and highest-value LLM upgrades.

Later, the LLM can:

- write a stronger analyst memo
- explain tradeoffs more clearly
- connect growth evidence and risk evidence in a more natural way

### 5. Better retry logic

Right now the loop is simple.

Later, the LLM can decide:

- if the evidence is still weak
- what kind of evidence is missing
- whether another retrieval step is worth it

## Simple Direction

Keep the system stable underneath, and add more LLM help on top of it step by step.

That means:

- keep retrieval and analysis trustworthy
- make interpretation and synthesis smarter over time
- avoid turning the whole system into a black box too early

## Goal

The next version of the system should be:

- more dynamic in how it routes questions and chooses tools
- more flexible across different analyst-style tasks
- still stable, inspectable, and grounded in retrieved evidence

In other words, the long-term goal is **LLM-guided orchestration on top of deterministic tools**, not an uncontrolled end-to-end black box.

## Where We Are Now

The current system is mostly a staged pipeline:

```text
User Question
-> Collector Agent
-> Retrieval
-> Evidence Bundle
-> EDA Agent
-> Analysis Findings
-> Analyst Agent
-> Final Answer
```

There is one non-linear part today:

```text
EDA Agent
-> if evidence is weak or incomplete
-> back to Collector Agent for one more retrieval pass
```

So the current implementation is best described as:

- **main path:** linear
- **current graph behavior:** linear plus one retry loop
- **LLM usage today:** light synthesis/refinement, not full control

## Current LLM-Driven Parts

Right now, only a small part of the system is LLM-driven.

### In the current codebase

- `graph/workflow.py::_build_research_plan()`
  - The routing itself is still heuristic.
  - The LLM only refines the research note text.

- `graph/workflow.py::_eda_node()`
  - EDA findings are computed deterministically.
  - The LLM can add one short takeaway note.

- `graph/workflow.py::_analyst_node()`
  - The LLM can write the final natural-language answer from the evidence and findings.

### Important constraint

The LLM currently does **not** control:

- retrieval choice
- SQL generation
- EDA tool selection
- retry logic
- multi-step planning

That means the system today is:

**deterministic core + optional LLM narration**

## Deterministic Now vs Dynamic Later

| Component | Current Behavior | Future Dynamic Upgrade |
|---|---|---|
| Question routing | Heuristic keyword classification in `graph/workflow.py` | LLM decides question type, subgoals, and whether to split into sub-questions |
| Research planning | Fixed mapping from category to goals/tools | LLM builds a plan dynamically and chooses which tools to call |
| Collector behavior | Deterministic retrieval rules | Agent chooses retrieval order, retries, and whether more evidence is needed |
| Qualitative retrieval | Keyword-based chunk search over filings and press releases | Embedding/vector retrieval, query rewriting, richer source selection |
| Quantitative retrieval | Fixed SQL-backed helper functions | Text2SQL or schema-aware query planning |
| EDA tool selection | Hardcoded by question category | EDA agent chooses tools based on evidence quality and task type |
| Financial EDA | Deterministic period-over-period calculations | Dynamic comparisons, anomaly detection, benchmarking |
| Market EDA | Deterministic return and volatility analysis | Smarter event windows, multi-period reaction analysis |
| Text EDA | Deterministic term and section counting | Theme extraction, clustering, contradiction detection |
| Retry logic | One fixed retry loop from EDA back to Collector | Multi-step iterative refinement with stopping conditions |
| Final answer | Deterministic grounding plus optional LLM prose | Richer synthesis, alternative hypotheses, self-critique |
| Charts/artifacts | Deterministic chart and memo generation | LLM chooses best visualization/report format for the question |

## Design Principle for the Next Version

We do **not** want to make everything LLM-driven at once.

That would make the system:

- harder to debug
- harder to trust
- harder to explain in the assignment/demo
- more likely to fail unpredictably

Instead, we should keep this separation:

### Keep deterministic

- database access
- core retrieval implementations
- numeric EDA calculations
- artifact writing
- schema validation
- fallback answer construction

### Make more dynamic gradually

- routing
- tool selection
- query planning
- iterative refinement
- final synthesis

This keeps the system stable while still making it more capable.

## What We Want the Future System to Look Like

The future system should be more graph-shaped than stage-shaped.

```text
User Question
-> Orchestrator
   -> Collector
      -> qualitative retrieval
      -> quantitative retrieval
   -> EDA
      -> financial analysis
      -> market analysis
      -> text analysis
   -> Analyst
   -> optional Critic / Verifier
-> Final grounded answer
```

Desired behavior:

- not every question should follow the exact same path
- qualitative-only questions should stay mostly on the text side
- quantitative-only questions should stay mostly on the metrics side
- mixed questions should combine both
- the system should be able to retrieve again if evidence is weak
- eventually, multiple analyses should be able to run in parallel

## Recommended Roadmap

## Phase 1: Keep the current system stable

Goal: strengthen the current implementation before adding too much autonomy.

### Tasks

- improve question classification rules
- improve metric selection logic in the Collector
- improve ranking/prioritization of financial findings
- improve final answer wording and evidence ordering
- add more tests for routing and EDA outputs
- make sure charts and artifacts are consistently produced

### Why this phase matters

This phase gives us a stronger baseline. If we add LLM control too early, it becomes hard to tell whether failures come from data quality, tool logic, or orchestration.

## Phase 2: Add dynamic routing

Goal: let the system choose paths more flexibly while keeping retrieval and analysis deterministic underneath.

### Tasks

- replace or augment heuristic routing with an LLM router
- let the router choose:
  - qualitative
  - quantitative
  - mixed
  - market-specific
  - narrative/risk-specific
- keep a deterministic fallback path when the LLM is unavailable

### Result

The system becomes more flexible without losing reliability.

## Phase 3: Add dynamic tool selection in EDA

Goal: let the EDA stage decide which analysis tools to run instead of using only hardcoded category-to-tool rules.

### Tasks

- give the EDA agent a shortlist of available tools
- let it choose among:
  - financial trend analysis
  - market reaction analysis
  - text theme analysis
  - chart generation
- keep each tool deterministic internally

### Result

The system becomes more adaptable to different question types while remaining auditable.

## Phase 4: Introduce iterative refinement

Goal: move from one fixed retry loop to a more intentional research loop.

### Tasks

- let EDA request more specific missing evidence
- let the Analyst request one more targeted retrieval pass when support is weak
- define stopping conditions such as:
  - enough evidence found
  - no new high-value evidence returned
  - max iterations reached

### Result

The system starts to behave more like a real analyst workflow instead of a single-pass pipeline.

## Phase 5: Add smarter quantitative querying

Goal: make structured data access more flexible.

### Recommended progression

1. Keep deterministic query helpers as the default
2. Add schema descriptions and glossary mapping
3. Add a controlled Text2SQL layer later

### Important note

The project does **not** currently use agent-generated SQL. The current SQL lives in deterministic helper functions under `storage/query_service.py`.

When Text2SQL is added, it should be:

- constrained
- validated
- optional
- backed by deterministic fallback tools

## Phase 6: Add richer qualitative retrieval

Goal: improve the quality of text evidence retrieval.

### Tasks

- keep keyword retrieval as a fallback
- add embeddings/vector search
- improve chunk ranking with metadata weighting
- possibly add query rewriting for vague analyst questions

### Result

Better retrieval quality without losing the ability to debug failures.

## Phase 7: Add a critic/verifier agent

Goal: make final answers more trustworthy.

### Tasks

- add a final verification step after the Analyst draft
- check whether:
  - claims are supported by evidence
  - sources are sufficient
  - contradictory evidence exists
- revise or downgrade confidence when support is weak

### Result

This gives the system a stronger “analyst + reviewer” structure.

## Suggested Future Agent Structure

### Current

- Collector Agent
- EDA Agent
- Analyst Agent

### Future

- Collector Agent
- Quantitative EDA Agent
- Qualitative EDA Agent
- Analyst Agent
- Critic / Verifier Agent

This would make the graph more modular and give each role a clearer responsibility.

## MCP Status

The repo currently has an `mcp_server/` folder, but it is empty and not used.

### Do we need MCP right now?

No.

For this project, MCP is not on the critical path. The current app already has:

- a frontend
- a LangGraph workflow
- internal tool calling
- a cloud deployment path

### When MCP would make sense later

MCP becomes useful if we want to expose the system's tools to:

- external clients
- other agent runtimes
- shared tool servers across apps

### Recommendation

- do not build MCP yet
- keep focusing on app quality, orchestration quality, and deployment

## Immediate Next Priorities

If we want the next iteration to be both stronger and safer, the best order is:

1. Improve current routing and output quality
2. Add tests around tool usage and stage outputs
3. Add dynamic routing with deterministic fallback
4. Add dynamic EDA tool selection
5. Add better qualitative retrieval
6. Add controlled iterative refinement
7. Add critic/verifier stage
8. Consider Text2SQL later, not first

## One-Sentence Direction

Build toward a **more agentic and flexible orchestration layer**, while keeping retrieval, EDA, and grounding logic deterministic enough to stay stable, explainable, and debuggable.
